# PRD-01 — Motor de Cálculo da Folha

---

## 1. Objetivo

Definir o funcionamento do motor de cálculo da folha de pagamento, garantindo:

- cálculo determinístico  
- execução baseada em fórmulas configuradas  
- ausência de heurísticas  
- rastreabilidade total dos valores  
- consistência entre folha, recibos e relatórios  

O motor é responsável por transformar os valores informados pelo usuário (rubricas operacionais) em resultados finais da folha, respeitando exclusivamente regras explícitas e configuráveis.

---

## 2. Princípios obrigatórios

- Nenhuma regra implícita  
- Zero heurística baseada em nome  
- Separação entre cálculo e exibição  
- Campos derivados não são entrada manual  
- O motor é a única fonte de verdade dos valores calculados  
- O sistema deve ser auditável  
- O motor NÃO pode sobrescrever valores informados manualmente  
- O sistema deve refletir a operação real da empresa  
- O cálculo deve ser definido por fórmulas configuradas  
- O sistema deve ser totalmente extensível sem alteração de código  

---

## 3. Papel do motor no sistema

O motor de cálculo é responsável por:

- executar fórmulas configuradas nas rubricas  
- resolver dependências entre rubricas  
- consolidar valores da folha  
- calcular resultados finais de forma determinística  
- garantir consistência entre todos os módulos  

O motor NÃO é responsável por:

- interface do usuário  
- definir regras fixas de cálculo  
- criar valores automaticamente sem configuração  
- sobrescrever valores manuais  
- impor lógica baseada em nome de rubrica  
- aplicar regras invisíveis ao usuário  

---

## 4. Modelo estrutural da folha

O sistema trabalha com dois tipos de dados:

---

### 4.1 Rubricas Operacionais (INPUT)

São os valores informados manualmente pelo usuário.

Exemplos:

- salario_ctps  
- salario_g  
- outros_rendimentos  
- horas_extras  
- salario_familia  
- ferias_terco  
- insalubridade  
- inss  
- emprestimos  
- vales  
- faltas  

Características:

- são editáveis  
- possuem classificação  
- são persistidas no sistema  
- são a única fonte de entrada da folha  
- podem ser alteradas a qualquer momento  

---

### 4.2 Rubricas Calculadas (OUTPUT DO MOTOR)

São valores gerados automaticamente pelo sistema com base em fórmulas configuradas.

Exemplos:

- salario_real  
- g2_complemento  
- salario_liquido  

Características:

- não são editáveis diretamente  
- não possuem classificação  
- são resultado da execução de fórmulas  
- dependem de outras rubricas  
- são reproduzíveis a qualquer momento  

---

## 5. Regra CRÍTICA — INSS

- INSS é uma rubrica operacional do tipo desconto  
- INSS deve ser informado manualmente pelo usuário  
- O motor NÃO deve calcular INSS automaticamente  
- O motor NÃO pode sobrescrever o valor informado  

Motivação:

> O sistema deve refletir a operação real da empresa.

---

## 6. Execução do motor de cálculo

---

### 6.1 Regra estrutural — Execução por Fórmula

O motor deve operar como um **executor de fórmulas configuradas**, e não como um conjunto de regras fixas.

---

### Princípio central

> O sistema NÃO define como calcular valores derivados.  
> O usuário define através das fórmulas cadastradas.

---

### Fluxo de execução

1. Carregar todas as rubricas operacionais
2. Identificar rubricas calculadas
3. Ordenar cálculo com base em dependências
4. Resolver dependências entre rubricas
5. Executar fórmulas
6. Gerar resultados finais

---

### 6.2 Execução de fórmulas

Para toda rubrica com:

- natureza = `calculada`
- método_calculo = `formula`

O motor deve:

- interpretar a fórmula cadastrada  
- substituir referências por valores reais  
- executar a expressão matemática  
- retornar o resultado  

---

### 6.3 Dependência entre rubricas

O sistema deve suportar:

- uso de múltiplas rubricas em uma fórmula  
- encadeamento de cálculos  
- dependências entre rubricas calculadas  
- ordem de cálculo dinâmica  

---

### 6.4 Ordem de cálculo

A ordem deve ser definida por:

- dependência entre rubricas  
- não por ordem fixa manual  

O sistema deve:

- detectar dependências automaticamente  
- impedir dependência circular  
- garantir execução consistente  

---

## 7. Regras críticas do motor

- não sobrescrever valores manuais  
- não ignorar fórmulas cadastradas  
- não hardcodar regras de cálculo  
- não inferir comportamento por nome  
- não misturar input com output  
- não permitir dependência circular  
- não aplicar lógica invisível  
- não executar cálculos fora da ordem de dependência  

---

## 8. Integração com rubricas (PRD-02)

- rubricas operacionais alimentam o motor  
- rubricas calculadas são executadas via fórmula  
- classificação NÃO interfere no cálculo  
- o motor NÃO deve criar rubricas automaticamente  
- o motor NÃO deve alterar valores operacionais  

---

## 9. Persistência de dados

- valores de entrada devem ser persistidos  
- fórmulas devem ser persistidas  
- valores calculados podem ser recalculados  
- o sistema deve permitir reprocessamento completo  

---

## 10. Auditoria

O sistema deve permitir:

- rastrear cada valor informado  
- identificar origem de cada rubrica  
- visualizar fórmula aplicada  
- reproduzir cálculo completo  
- validar resultados com transparência  

---

## 11. Segurança

- cálculo não pode depender da interface  
- backend é responsável pela execução das fórmulas  
- não permitir manipulação manual de valores calculados  
- validação de fórmulas deve ser obrigatória  

---

## 12. Performance

- cálculo deve ser rápido (< 200ms por funcionário)  
- permitir execução em lote  
- evitar recomputação desnecessária  
- otimizar resolução de dependências  

---

## 13. Estados do cálculo

### Sucesso
- cálculo completo e consistente  

### Incompleto
- ausência de dados necessários  

### Erro
- fórmula inválida  
- dependência circular  
- referência inexistente  

---

## 14. Estratégia de implementação

1. receber rubricas operacionais  
2. carregar fórmulas configuradas  
3. montar grafo de dependências  
4. ordenar execução  
5. executar fórmulas  
6. gerar resultados finais  
7. retornar valores  

---

## 15. Fora do escopo

- cálculo automático de impostos  
- regras trabalhistas complexas  
- integração com governo  
- validações fiscais avançadas  
- múltiplos motores paralelos  

---

## 16. Critérios de sucesso

- usuário controla totalmente as regras de cálculo  
- sistema não depende de lógica hardcoded  
- novas regras podem ser criadas sem desenvolvimento  
- cálculo é consistente e previsível  
- resultados batem com o sistema atual da empresa  
- ausência total de heurística  
- confiança total no sistema
